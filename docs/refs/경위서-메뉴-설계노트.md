# 경위서(Incident Report) 메뉴 — 설계 스펙

> 상태: **설계 확정** (사용자 승인 완료 2026-06-01) — 구현 전.
> 단계: **1차 = 작성·승인·PDF/Word·메일** / **2차 = SharePoint 연동**

## 1. 목표

자료보관 > 경위서 화면. 운영자가 **사고(incident) 1건에서 경위서를 생성**하여
4섹션 내용을 다듬고, 팀장 승인을 거쳐 고객(대학 담당자)에게 PDF로 메일 발송한다.
2차로 생성 Word를 SharePoint `운영부 > 06. 경위서` 폴더에 운영자 본인 이름으로 업로드하고,
시행번호는 `10. 공문 > 공문관리대장.xlsx`와 자동 연동한다.

핵심: 경위서 = 사고 위에 얹는 **"문서화 → 결재 → 발송" 레이어**. 내용 재입력 없음.

## 2. 양식 구조 (샘플 5건 분석)

경위서 = **2장 세트**.

### ① 공문 (cover, 정형 보일러플레이트)
- 머리말(회사 슬로건 고정) / 수신자(대학명) / 참조 / 제목
- 인사말 "귀교의 무궁한 발전을 기원합니다" + 사과 본문 + 붙임 "~경위서 1부. 끝."
- **결재라인**: `담당자 → 팀장 → 본부장 → 사장` (전결 표기)
- 시행번호 `서비스사업YYMM-NNNN(YYYY.MM.DD)` + 회사 주소/연락처(고정)

### ② 경위서 본문
- 작성일자 / 작성자 / 제목
- **4섹션** (제목 표현은 케이스별 편집 가능):
  1. 경위 (=내용/사고내용)
  2. 원인 (=사고원인)
  3. 처리 — 시간/내용 2열 타임라인 (1차는 text, 구조화 표는 추후)
  4. 향후 대책
- 맺음말 사과 문구

## 3. 데이터 매핑 (incidents → incident_reports)

| 경위서 | incidents 필드 |
|---|---|
| 수신대학 | `university_name` |
| 제목 | `title` |
| 1. 경위 | `cause_summary` |
| 2. 원인 | `root_cause` |
| 3. 처리 | `resolution` |
| 4. 대책 | `prevention` |
| 작성자(담당자) | `assignee_name` / `assignee_email` |

## 4. 확정 결정사항

| 항목 | 결정 |
|------|------|
| 도메인 | 독립 도메인(자료보관 > 경위서), incidents에 **선택적 FK** 연결. 연결 시 4섹션 자동 프리필 |
| 섹션 구조 | 4섹션 기본 + 섹션 제목 편집 가능 (추후 표 구조화) |
| 결재라인 | 담당자=작성자, 팀장=operator `leader` 자동, **본부장·사장=operators `role` enum 확장** 후 자동 |
| 승인자 | 작성자 operator의 소속 팀 **팀장 자동** (`leader`) |
| 승인 방식 | **앱 내 팀장 승인** (팀장 로그인 → 승인 대기 목록) |
| 출력 | **Word + PDF 둘 다** (`docx` npm 신규) |
| 고객 메일 | **contacts 도메인에서 선택** (university_name 필터 → contact_email) |
| 시행번호 | **공문관리대장.xlsx 자동 읽기 + 행 추가** (2차) |
| SharePoint 업로드 | 06.경위서 폴더, **"만든 사람"=운영자 본인 표시** (2차, spike 필요) |

## 5. 상태·승인 흐름

```
draft ──제출──▶ pending_approval ──팀장승인──▶ approved ──발송──▶ sent
   ▲────────────반려(사유)───────────┘
```
- 팀장 = 작성자 `leader` 자동 배정. 팀장 로그인 시 "승인 대기" 목록 노출.
- 발송은 approved 이후 작성자/admin만.

## 6. 컴포넌트 설계

### 1차 범위
- **DB 마이그레이션**
  - `incident_reports` 테이블 (위 스냅샷 필드 + status + 결재라인 + 승인 메타) + RLS + GRANT
  - `incident_report_mail_sends` 이력 테이블
  - operators `role` enum 확장: `본부장`, `사장` 추가
- **features/incident-reports/** — schemas / queries / actions(create/update/submit/approve/reject) / mail-actions / mail-template (+ __tests__)
- **문서 생성**
  - `lib/pdf/incident-report-pdf.tsx` (@react-pdf, 2장: 공문+경위서, Pretendard, 결재라인 표)
  - `lib/docx/incident-report-docx.ts` (`docx` npm 신규) — 동일 2장 → Buffer
- **메일** — `sendGraphMail` 재사용(운영자 메일박스), PDF 첨부, 수신=contacts 선택, `MAIL_DRY_RUN` 준수
- **UI** — list-variant `incident-reports` (registry 1줄 + types union 1줄 + View/EditForm/Table/filters) + 자료보관 사이드바 메뉴 등록. 진입: incidents "경위서 생성" or 경위서 메뉴 신규(사고 선택)
- **결재라인 해석** — operator `leader` → 팀장, role=본부장/사장 → 임원 자동

### 2차 범위 (SharePoint)
- 공문관리대장.xlsx 읽기+행추가 (workbook-session 패턴, env: `SHAREPOINT_GONGMUN_*`)
- 06.경위서 폴더 .docx 업로드 — Graph driveItem PUT 신규 (env: `SHAREPOINT_INCIDENT_REPORT_*`)
- **작성자 메타 = 운영자 본인** → **위임(delegated) 인증 확장**으로 해결 (정공법):
  - 현 상태: `login/page.tsx`가 `signInWithOAuth({provider:"azure"})`로 MS SSO 작동 중. 단 스코프가 `openid profile email`뿐(신원용)이고 `auth/callback`이 `provider_token`/`provider_refresh_token`을 버림
  - 확장: (1) SSO 스코프에 `Files.ReadWrite.All` `Sites.ReadWrite.All` `offline_access` 추가 (2) 콜백에서 `provider_refresh_token` 캡처·저장(업로드는 발송 시점 → refresh token 필요) (3) Azure AD 앱에 위임 권한 + admin consent (4) 위임 토큰으로 driveItem PUT → "만든 사람" 자연히 운영자
  - 폴백(위임 불가 시): 서비스 계정 업로드 + 파일명/본문에 작성자 명시
  - 주의: Supabase는 `provider_*` 토큰을 sign-in 직후 1회만 노출 → 캡처·영구저장 필수, refresh token 회전(rotation) 처리 필요

## 7. 권한 / RLS

- read: authenticated 전체 (운영부 공개)
- write(작성/수정): member 이상, 본인 draft
- 승인: 해당 경위서 팀장 본인만
- 발송: 작성자 또는 admin
