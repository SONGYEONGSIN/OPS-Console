# 경위서 → 사고보고 통합 설계

> 작성일: 2026-06-02
> 상태: **설계 승인됨** (사용자 확인) — 구현 전
> 브랜치: `feat/incident-report-merge-into-incidents`

## 1. 목표

경위서(incident-reports) 작성·결재·발송 프로세스를 **사고보고(incidents) 메뉴 안으로 통합**한다.
두 메뉴는 성격이 유사하고(원래 설계 노트에서도 "경위서 = 사고 위에 얹는 문서화→결재→발송 레이어"),
현재는 별도 메뉴 + 느슨한 FK 연결 상태다. 이를 **단일 메뉴(사고보고) + 인스펙터 탭** 구조로 합친다.

### 핵심 전제
- **경위서 데이터 현재 0건** (기능은 #304/#312로 구현됐으나 실사용 전), 사고 756건 → 마이그레이션 무손실, 클린 재편 가능
- 사고 1건 ↔ 경위서 **0..1** (선택). 사고를 보다가 필요하면 그 자리에서 경위서를 만든다.

## 2. 범위 (In / Out)

### In scope
- 사고 인스펙터에 `[사고정보 | 경위서]` 탭 추가, 경위서 워크플로우를 탭 안에서 처리
- 기존 `incident-reports` list-variant 컴포넌트(View/EditForm)를 탭에서 **재사용**
- 사고 목록 행에 경위서 상태 배지
- 사고보고 목록에 `경위서 승인 대기` 스코프 칩 (팀장 결재 큐 대체)
- 사이드바 「자료 보관 > 경위서」 메뉴 제거 + `/dashboard/incident-reports` → `/dashboard/incidents` 리다이렉트
- DB: `incident_reports.incident_id` NOT NULL + UNIQUE

### Out of scope (변경 없음)
- 경위서 액션 권한 로직(작성=로그인, 발송=작성자/admin, 승인=결재자) — 그대로
- 경위서 PDF/docx/메일/SharePoint 파이프라인 내부 로직 — 호출 위치만 이동
- 비정규화 컬럼 추가 안 함 (경위서 상태는 `incident_id` 조인으로 도출)

## 3. 데이터 모델

```
incidents (1) ──── (0..1) incident_reports
                   FK: incident_reports.incident_id → incidents.id
                   제약: NOT NULL + UNIQUE (사고당 경위서 최대 1건)
```

### 마이그레이션 (`supabase/migrations/20260602e_incident_reports_incident_required.sql`)
```sql
begin;
-- 데이터 0건 전제. 혹시 standalone(incident_id null) 있으면 마이그 실패하도록 가드.
alter table public.incident_reports
  alter column incident_id set not null;
alter table public.incident_reports
  add constraint incident_reports_incident_id_unique unique (incident_id);
notify pgrst, 'reload schema';
commit;
```
- 적용 전 `select count(*) from incident_reports where incident_id is null` = 0 확인 (선적용 워크플로우 준수)

## 4. UI 설계 (인스펙터 탭)

### 4.1 탭 구조
- `IncidentView`(읽기 모드)에 탭 상태 `useState<"info" | "report">("info")`
- **사고정보 탭**: 기존 4섹션(경위/원인/처리/대책) 등 현행 유지. 사고 수정은 기존 Edit 모드 진입
- **경위서 탭**:
  - 연결 경위서 **없음** → `경위서 작성` 버튼. 클릭 시 `onCreateReport(incidentId)` → 사고의 cause_summary/root_cause/resolution/prevention 자동 prefill로 draft 생성
  - 연결 경위서 **있음** → 기존 `IncidentReportView`(상태별 승인요청/승인/반려/발송 UI) + 편집 진입. 컴포넌트 재사용

### 4.2 탭 vs 편집 모드 경계
- 탭(사고정보/경위서)은 **읽기 모드 레이어**에만 존재
- 사고 자체 편집은 사고정보 탭의 기존 수정 버튼 → 인스펙터 Edit 모드(경위서 탭 비노출)
- 경위서 편집/워크플로우는 경위서 탭 내부에서 자체 상태로 처리 (사고 Edit 모드와 독립)

### 4.3 목록 배지
- `incidents/Table.tsx` 상태 컬럼 뒤에 **경위서 상태 배지** 컬럼: 없음 / 작성중(draft·rejected) / 승인대기 / 승인 / 발송
- 라벨·색상은 기존 `incident-reports` status 상수 재사용

## 5. 서버 액션 주입 (인스펙터 props 확장)

현재 인스펙터 변형 컴포넌트는 registry 정적 바인딩. 경위서 워크플로우를 사고 탭에서 구동하려면
경위서 데이터 + 서버 액션을 `incidents/page.tsx`에서 주입해야 한다.

### `ViewProps` 확장 (`list-variants/types.ts`)
```ts
// 추가 (모두 optional — incidents variant에서만 사용)
incidentReport?: IncidentReportRow | null;          // 연결된 경위서(0..1)
incidentReportRecipients?: RecipientCandidate[];    // 발송 수신 후보
incidentReportApprovalChain?: ApprovalChain;        // 결재선
onCreateReport?: (incidentId: string) => Promise<ActionResult>;
onSubmitReport?: (id: string) => Promise<ActionResult>;
onApproveReport?: (id: string) => Promise<ActionResult>;
onRejectReport?: (id: string, reason: string) => Promise<ActionResult>;
onSendReport?: (input: { id: string; recipient_emails: string[] }) => Promise<ActionResult>;
```

### `incidents/page.tsx` 배선
- 선택된 사고에 대해 `getIncidentReportByIncidentId(incidentId)` 조회 (신규 쿼리 — `incident_id` 기준)
- 경위서 존재 시 `listRecipientCandidates(university)` + `resolveApprovalChain(author)` 로드
- 위 서버 액션들을 `"use server"`로 정의해 ViewProps에 주입 (기존 `incident-reports/page.tsx` 배선 이전)
- 주: 인스펙터는 목록 전체를 들고 있으므로, 데이터 로딩은 "선택 시 lazy" 또는 "행마다 경위서 상태만 batch 조회 + 상세는 선택 시" 중 **batch 상태 + lazy 상세** 채택 (목록 N+1 회피)

## 6. 승인 큐 (팀장 결재)

경위서 메뉴를 없애면 팀장의 "승인 대기 경위서" 화면이 사라진다. 대체:
- 사고보고 목록에 스코프 칩 `경위서 승인 대기` 추가 (`ScopeChips` 확장 또는 별도 칩)
- 필터: `incident_reports.status = 'pending_approval' AND approver_email = me.email` 인 경위서가 달린 사고만
- 구현: `incidents/page.tsx`에서 쿼리 파라미터(`?report=pending`) 기반 필터링

## 8. 권한 모델 (현행 유지)

| 액션 | 권한 | 비고 |
|---|---|---|
| 사고 작성/수정 | 로그인(member↑) | 기존 |
| 경위서 작성 | 로그인 | incident_id 필수로 변경 |
| 경위서 승인요청 | 작성자 or admin | 기존 |
| 경위서 승인/반려 | 결재자(approver_email=팀장)만 | 기존 |
| 경위서 발송 | 작성자 or admin | 기존 |

호출 위치만 인스펙터로 이동, 액션 내부 가드는 불변.

## 9. 에러 처리

- 경위서 작성 시 사고 미연결 불가(incident_id 필수) — 항상 사고 컨텍스트에서 생성되므로 자연 충족
- 발송: `getDelegatedGraphToken` 실패 / SharePoint env 미설정 / `MAIL_DRY_RUN` — 기존 mail-actions 가드 유지, 인스펙터에 에러 메시지 표시(기존 `run()` 패턴)
- 동시성: 이미 `pending_approval`/`sent`인 경위서 재전이 방지(기존 상태 가드 유지)
- 마이그레이션: standalone 경위서(0건 전제 위반) 존재 시 NOT NULL 적용 실패로 안전 차단

## 10. 테스트 전략 (TDD)

- **단위**
  - `IncidentView` 탭 전환(info↔report), 경위서 없음/있음 분기 렌더
  - 경위서 상태 배지 매핑(없음/작성중/승인대기/승인/발송)
  - `경위서 승인 대기` 필터 로직(approver=me & pending)
  - prefill 매핑(사고 4필드 → 경위서)
- **기존 유지**: `incident-reports` 액션 테스트(권한·상태전이) 그대로, 호출부 이동분 추가
- **E2E 스모크 1건**: 사고 선택 → 경위서 탭 → 작성 → 승인요청까지

## 13. 미해결 / 리스크

- 인스펙터가 목록 전체를 들고 있어 경위서 상세를 "선택 시 lazy 로드"하는 메커니즘이 현재 인스펙터 패턴에 없을 수 있음 → 구현 1순위 확인 사항. 최악의 경우 목록 로드시 batch 상태만, 상세 액션은 선택된 행 기준 서버 액션 호출로 처리.
- `ScopeChips`는 현재 `?mine=` 단일 토글 → `경위서 승인 대기`는 별도 칩/쿼리파라미터로 확장 필요.
