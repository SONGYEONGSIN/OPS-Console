# Brainstorm: 분석보고서 메뉴 — /dashboard/reports

생성 2026-05-25T19:34:12+09:00 · 주제: 분석 & 보고 그룹의 placeholder `reports` 메뉴 화면 구성

## 의도

- **무엇을**: `/dashboard/reports` 메뉴 — 운영부 누적 데이터의 **기간별 집계 KPI 대시보드** + **저장된 리포트 목록**(1차 placeholder). 동일 그룹의 `worklog`(raw event)·`outcomes`(개인 8단계 평가)와 차별화된 **조직·도메인 단위 트렌드**.
- **누가**:
  - 운영부 전체 (admin/member) — 정기적 운영 현황 점검
  - 임원·관리자 — 분기·월간 운영 종합 파악
  - 시점: 매월 마감·분기 회고·신규 운영자 온보딩 시 운영 규모 학습
- **왜 지금**: 메뉴는 사이드바에 등록되어 있으나 placeholder 상태(`[slug]` dynamic fallback). 운영부가 자주 마주치는 "이번 달 사고가 늘었나? 미수채권은 줄었나?" 질문에 답이 없음. 같은 그룹의 worklog는 raw, outcomes는 개인 단위 — 조직 단위 집계 공백.
- **성공**:
  - 메뉴 진입 1번 클릭 → 기간 selector + 8 KPI 카드 + 리포트 목록
  - 기간 변경 시 즉시 카드 수치·전 기간 대비 % 갱신
  - 카드 수치는 실측 (services/incidents/contracts/receivables/handover_progress/backup_requests/mail_sends/worklog DB 직접 조회)
  - 리포트 목록은 1차 placeholder (2~3 mock entries) — 사용자가 후속에서 실제 생성 흐름 결정

## 제약

**기술**:
- 데이터 source 8개 — Supabase(services/incidents/handover_progress/backup_requests/worklog/3 mail_sends) + SharePoint(contracts/receivables)
- 기간 비교 → 현재 기간 vs 직전 동기간(이번 달 vs 지난 달, 분기 vs 전 분기 등)
- SharePoint 호출 비용 — contracts/receivables 시트 전체 fetch 후 client filter. 캐시 60s 권장
- HARD-GATE 등급: 영향 파일 ~10개 → **간략 설계**

**비즈니스**:
- 1차 MVP — KPI 8 카드 + 리포트 목록 placeholder
- 리포트 자동 생성·PDF 출력은 follow-up (사용자 명시)
- viewer 권한도 read 가능 (조직 통계는 공유)

**코드베이스**:
- settings/operating-guide/manuals 페이지와 동일 톤 (PageHeader + section + 자체 KPI grid)
- ListPattern 미사용 — KPI 카드는 별도 컴포넌트, 리포트 목록은 단순 list

## 추천 + 근거

**추천**: **대안 A — 결합형 (KPI 대시보드 + 리포트 목록 placeholder)**

**선택 근거**:
1. 사용자가 명시적으로 A + C 혼합 원함 — 한 화면에서 즉시 overview + 향후 산출물 입구
2. 리포트 목록은 placeholder만으로도 향후 확장 의도 시각화 — "이런 리포트가 생길 것이다" 안내
3. 1차 MVP 비용은 7 vs 10 파일로 차이 미미

**기각된 대안 B**: 대시보드만으로는 "분석보고서"라는 메뉴명과 맞지 않음. "보고서" 자체가 생성 가능한 산출물을 함의 — placeholder라도 그 영역이 있어야 메뉴명 정합.

**기각된 대안 Z**: 사이드바에 메뉴 있고 사용자가 클릭하면 빈 placeholder. 메뉴 신뢰도 ↓. 1차 MVP 비용 적음.
