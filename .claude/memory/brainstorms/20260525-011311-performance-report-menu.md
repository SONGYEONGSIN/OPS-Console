# Brainstorm: 성과리포트 메뉴 화면 구성

## 의도
- **산출물**: `/dashboard/performance` 메뉴. 8단계 평가 워크플로우(목표설정 → 실행계획 → 계획검토 → 중간점검 → 점검검토 → 자기평가 → 종합평가 → 완료) + 평가 항목 누적 관리. 평가자(팀장/부장)와 팀원(운영자) 권한별 액션 분리.
- **사용자**: 
  - 평가자: 목표 부여 / 계획 검토 / 점검 검토 / 종합평가 입력
  - 팀원(운영자): 본인 실행계획 / 중간점검 / 자기평가 작성
  - 임원/대외(읽기): 누적 평가 조회 (PDF 다운로드는 follow-up)
  - 시점: 시즌 마감 / 분기 평가 주기 + 상시 자기 확인
- **트리거**: 관리자가 시스템 한곳에서 평가 누적 관리·평가 수행. 현재 외부 도구(엑셀/notion) 분산 → 일원화 필요.
- **성공 기준**:
  1. 8단계 워크플로우가 한 메뉴 안에서 완주 (각 단계 권한별 트랜지션)
  2. 평가 항목 80%(성과지표 = 원서접수+PIMS 40% / 차세대 20% / 개인활동 20%) + 20%(관리자지표) 가중 계산 자동
  3. 연차별 다른 기준 적용 가능 (시즌·연도별 평가 템플릿)
  4. 평가자/팀원 권한 가드 — 본인 평가만 작성, 팀장은 부하 직원만 검토
  5. 누적 이력 조회 (연도/평가 사이클별)

## 제약
- **기술**: 
  - Next.js App Router + Supabase + zod. 신규 도메인 (`performance_cycles`, `performance_goals`, `performance_plans`, `performance_reviews`, `performance_scores`).
  - RLS 정책 필수 (본인/평가자만 INSERT·UPDATE 가능). 권한 매트릭스 = 단계 × role.
  - 가중치 계산은 server-side 집계 (CR 시 일관).
- **비즈니스**: 
  - 8단계 워크플로우는 사람 트랜지션 — 한 단계 완료 후 다음 단계 활성. 단계별 작성자/검토자 명확.
  - 연차별 기준 변경 → 평가 템플릿 버전 관리 필요.
  - 1차 PR은 골격 + 단일 시즌(2026) 검증. PDF 출력 / 연차별 템플릿 / 임원 뷰는 follow-up.
- **코드베이스**: 
  - 기존 인스펙터 + ListPattern 재사용 — 평가 사이클 1건 = ListRow 1건, 인스펙터 본문이 8단계 stepper.
  - operators 테이블의 role/team 활용 (평가자/평가받는 사람 매핑).
  - `worklog`/`incidents`/`handover_records` 등에서 개인 활동지표 자동 집계 후 평가 입력에 prefill (follow-up).

## 추천 + 근거

### 추천: 대안 A — 8단계 stepper + 단일 평가 카드

**선택 근거**:
1. 사용자가 직접 제시한 1번째 이미지(8단계 stepper)가 명확한 의도 신호 — 그대로 구현이 사용자 멘탈모델과 정합.
2. 한 평가 사이클의 모든 정보(목표 / 계획 / 점검 / 평가)가 한 화면에 집결 → 관리자가 누적 진행 상태 즉시 파악. 트리거("관리자가 한곳에서 누적 관리") 직접 해소.
3. 권한별 액션은 stepper 단계에 따라 자동 노출/잠금 (예: 3단계 계획검토는 평가자만 활성). 단계 분리 메뉴(C)보다 일관성·발견성 ↑.
4. 인스펙터 패턴 재사용 가능 — 평가 사이클 1건 = ListRow 1건, 인스펙터 본문이 stepper. 신규 컴포넌트 부담 ↓.

**기각된 대안 B**: 8단계 흐름이 사용자가 명시한 핵심인데 stepper 없이 폼만 두면 단계 인지 어려움. "어디까지 진행됐는지" 시각 단서 부재 → 평가자가 누락 검토 위험.

**기각된 대안 C**: 메뉴 2개 분리는 데이터 모델은 동일한데 UI만 다름 → 유지보수 비용 2배. 단일 메뉴 + 권한별 다른 액션이 더 깔끔.

**B로 전환 가치 있는 상황**: 8단계가 실제 운영에서 너무 무겁다고 판단되어 "목표 → 평가" 2단계로 단순화하기로 결정될 때.
