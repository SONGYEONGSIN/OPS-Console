# Brainstorm: MenuBar 상단 chrome 재구성 — 검색창 + 사용자 정보 영역

## 의도

- **산출물**: dashboard 모든 라우트가 공유하는 MenuBar 상단 chrome 재구성. 검색창의 비중/위치, 사용자 정보 표현, 좌측 ◆ 마커의 의미 명확화.
- **사용자**: 운영부 17명 (1차 송영신/2차 한효진 등). 시점은 (1) /dashboard 진입 직후 1면을 스캔할 때, (2) 47메뉴 페이지에서 작업 중일 때. 대체 행동은 사이드바로 메뉴 이동 또는 URL 직접 입력.
- **트리거**: 1면(/dashboard index) 신문 메타포 정착 직후. chrome이 본문(Lede/ProjectGrid/Activity)과 톤 충돌. 검색창이 1차 시야 240px 점유로 가독성 저하. 다음 분기로 미루면 47 메뉴 디테일 작업 진행 중 chrome이 점점 어색해진다.
- **성공 기준**:
  - 1면 진입 시 chrome 시각 부담 < 본문 (스크린샷 테스트로 ProjectGrid·Lede가 시야 60% 이상 점유)
  - power user는 ⌘K 단축키로 즉시 검색 가능 (검색 자체 기능은 보존)
  - 좌측 ◆ 클릭/주변에 "여기 어디" 컨텍스트 명확
  - 47 메뉴 페이지에서도 동일 chrome 유지 (route-aware 분기 도입 X — 일관성)

## 제약

- **기술**:
  - `layout.tsx`가 모든 dashboard 하위 라우트 공유 — chrome 일관성 강제.
  - 검색 backend 미연동 (placeholder만). 검색 기능 자체는 별도 epic.
  - design-tokens (vermilion/sage/ink/washi) 사용 강제 — 하드코딩 색상 금지.
- **비즈니스**:
  - 운영부 도메인 17명만 사용 → 검색이 외부 user-facing 핵심 도구가 아님.
  - 1면 정착 PR과 같은 사이클 — 큰 공사 X.
- **코드베이스**:
  - `MenuBar.tsx` 단일 파일에 ◆ + flex-1 + MenuRight (검색/알림/사용자) 구조.
  - 모바일은 `AppBar` 별도 (햄버거 + 알림 + 송영석) — 본 brainstorm 범위는 데스크탑 한정.

## 대안 비교

### 대안 A: 검색 축소 (icon-only)
- 핵심 아이디어: 검색창 240px → ⌘K 아이콘 1개로 압축, 클릭/단축키로 모달 펼침. 좌측 ◆ → "Folio · 실시간 현황" 컨텍스트 라벨.
- 비용: ~30분 (MenuBar.tsx + 작은 모달 placeholder)
- 위험: 검색 모달 미완성 시 클릭해도 빈 상태 노출 (placeholder text로 안내 가능)
- 가역성: 매우 쉬움 (CSS+JSX 재배치)
- 학습 효과: 1면 chrome 디자인 원칙 (본문 우선 / power user 단축키 보존)

### 대안 B: 검색 유지 + 좌측 보강
- 핵심 아이디어: 검색창 240px 그대로, 좌측 ◆ 옆에 "Folio · 실시간 현황" 또는 "OPSROOM" 워드마크 추가.
- 비용: ~10분
- 위험: 검색창 비중은 그대로라 1면 본문 시야 점유 미해결. 좌측 워드마크가 사이드바 그룹 헤더와 의미 중복 가능.
- 가역성: 매우 쉬움
- 학습 효과: 적음 (현 패턴 유지)

### 대안 Z: 아무것도 안 하기
- 지금 안 하면 47메뉴 디테일 작업 진행될수록 chrome 부담이 본문 가독성을 침범. 1면이 신문 메타포라는 정체성과 chrome 톤 충돌이 누적.
- 임시 우회책 없음 — chrome은 모든 라우트 공유라 페이지별 처리 불가.

## 추천 + 근거

### 추천: 대안 A — 검색 축소 (icon-only)

**선택 근거**:
1. 1면이 "읽기 전용 신문" 메타포 — 검색은 부차 도구. 240px → 아이콘으로 압축하면 본문이 시야 점유.
2. ⌘K 단축키는 그대로 살아있어 power user 손실 없음 (운영부 17명 중 키보드 익숙한 사용자에 친화적).
3. 좌측 ◆ 옆 컨텍스트 라벨이 "여기 어디인지" 즉시 보여주는 보조 navigation — 47 메뉴를 오갈 때 도움.
4. 검색 backend 미구현 시점에서 240px placeholder를 노출하는 것은 시각 부담 대비 가치 낮음.

**기각된 대안 B**: 검색창 비중 미해결로 1면 본문 시야 충돌이 그대로. 좌측 워드마크는 사이드바 헤더와 의미 중복. 검색 backend가 완성되어 결과 미리보기를 보여주는 경우라면 B로 전환할 가치 있음 (지금은 X).

**기각된 대안 Z**: 47메뉴 디테일 작업 진행될수록 부채 누적. 지금 ~30분 투자가 향후 모든 페이지 작업의 가독성 baseline 설정.
