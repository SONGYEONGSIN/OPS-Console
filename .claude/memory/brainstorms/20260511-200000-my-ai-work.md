## 의도

- **산출물**: 운영자가 AI 활용 작업 결과물을 등록·공유하는 페이지 1개 (목록 + 인스펙터 + 편집). dashboard 동적 `[slug]` 라우팅 활용
- **사용자**:
  - 등록자 — 본인의 AI 활용 사례를 기록 (시간 절감 가시화 + 자기 회고)
  - 열람자 — 다른 운영자의 **재사용 프롬프트**를 복붙해 자기 업무에 적용
  - 시점 — 일/주 단위 작업 마무리 직후, 또는 동료가 비슷한 일 한다고 들었을 때
- **트리거**: 운영자별 AI 활용이 늘고 있으나 노하우가 사람에 갇혀 있음. 지금 안 하면 각자 같은 프롬프트를 다시 만든다 (중복 비용)
- **성공 기준**:
  - 첫 2주 5명 이상이 1건 이상 등록
  - 재사용 프롬프트 "복사" 클릭 이벤트 발생 (정량 검증은 follow-up)
  - 사이드바 count 하드코드 → 동적 동기

## 제약

### 기술
- Next.js App Router + Supabase RLS — 기존 패턴 그대로
- ListPattern variant 8개 도달 (부채). **default variant 재활용**으로 회피, 9개째 도달 후 컴포지션 추출 PR 별도
- Supabase 마이그레이션 운영 함정 메모리 적용 (RLS+GRANT 동시 / PostgREST cache reload)

### 비즈니스
- 한국어 UI 전용, 한자 금지
- 공개 범위 단순화 — **운영부 전체 공개 고정** (사용자 결정). private/team 분기 없음

### 코드베이스
- `src/features/<domain>/` 패턴 (schemas / actions / queries 분리)
- 동적 `[slug]` 페이지 + `findSidebarMeta` 매핑 활용
- design-tokens.ts + Tailwind 클래스만, 하드코딩 색상 금지

## 추천 + 근거

**대안 A 채택**.

- **선택 근거**:
  1. 핵심 가치가 "재사용 프롬프트 공유"라면 default가 공개여야 가치 실현 (private 디폴트는 자기검열 유발)
  2. ListPattern 부채는 9개째 도달 시 별도 PR로 정리 — 이번 epic 일정 지연 회피
  3. zod 날짜 cutoff 강제 시 예외 마찰만 늘어남, UI 안내문구로 충분
- **B 기각**: 권한 3단계는 9 운영자 규모에서 과한 분기. 추후 외부 노출 검토 시점에 재도입 가능
- **Z 기각**: 측정 불가능 + Folio가 "운영부 단일 진실" 지향이므로 외부 도구로 빼면 일관성 깨짐

## RLS 정책

```sql
-- SELECT: 인증된 운영자 전원
create policy "ai_work_select_all" on ai_work
  for select to authenticated using (true);

-- INSERT: 본인 author만
create policy "ai_work_insert_self" on ai_work
  for insert to authenticated
  with check (author_operator_id = (select id from operators where auth_user_id = auth.uid()));

-- UPDATE/DELETE: 본인만
create policy "ai_work_modify_self" on ai_work
  for update/delete to authenticated
  using (author_operator_id = (select id from operators where auth_user_id = auth.uid()));
```

GRANT `select, insert, update, delete on ai_work to authenticated` 누락 금지 (이전 마이그레이션 함정 메모리).
