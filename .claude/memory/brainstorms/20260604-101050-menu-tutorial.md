# Brainstorm: 메뉴별 상세 튜토리얼

현재 전역 5스텝 첫 방문 투어(driver.js)를 확장 — 권한으로 보이는 각 메뉴마다 ①메뉴 설명 ②리스트 행 클릭 → 인스펙터가 열리는 인터랙션 안내 ③메뉴별 버튼 기능 설명.

## 의도
- **산출물**: 메뉴(slug)별 상세 워크스루 투어. 각 메뉴 3종 설명(개요 / 리스트→인스펙터 인터랙션 / 버튼 기능). 권한으로 보이는 메뉴만 대상, 전체 메뉴 커버.
- **사용자**: 신규 운영부 구성원 + 특정 메뉴를 처음 여는 사용자. 현재 대체 행동 = 전역 5스텝 오리엔테이션만 보고 메뉴별 사용법은 직접 더듬음.
- **트리거(왜 지금)**: 튜토리얼을 막 도입(#316). 메뉴별 상세화 요청. 미루면 신규 구성원 온보딩 마찰(어떤 버튼이 뭔지 모름) 지속.
- **성공 기준**: 권한으로 보이는 각 메뉴에서 (개요+인터랙션+버튼) 스텝이 동작 + 온디맨드 재생 가능 + 권한 없는 메뉴는 스텝 미생성. 콘텐츠 누락 메뉴 0.

## 제약
- **기술**: driver.js 1.4 기반(기존). 대시보드는 단일 셸 — 사이드바 클릭 시 가운데에 list 열림, 인스펙터는 행 선택 시에만 표시. → 인스펙터 버튼을 스포트라이트하려면 "행 선택 상태"가 필요(자동 조작 vs 설명만의 분기점).
- **코드베이스**: 16 list-variant 각각 버튼 구성 상이(View/EditForm/Table). 권한 메뉴는 이미 `sections`로 필터링되어 사이드바에 전달됨 → 투어가 동일 데이터 소비 가능. 사이드바 항목에 `slug` 존재(앵커 추가 용이).
- **비즈니스**: 진짜 비용은 코드가 아니라 **전체 메뉴 × 3종 + 16 variant 버튼 인벤토리 콘텐츠 작성**. UI 변경 시 셀렉터/문구 동반 관리 필요(부채).

## 대안 비교

### 대안 A: 온디맨드 메뉴별 가이드 (스포트라이트 + 설명, 자동 조작 X)
- 핵심: 각 페이지 헤더에 '가이드' 버튼. 누르면 그 메뉴 전용 driver.js 시퀀스 — 목록 영역 스포트라이트("여기가 목록") → "행을 클릭하면 오른쪽 인스펙터가 열립니다"(설명만) → 인스펙터/버튼 영역 스포트라이트 + 기능 설명. 콘텐츠는 `tutorial-menu-copy.ts` slug별 사전.
- 비용: 중(인프라 소) + 대(콘텐츠). 위험: 낮음(실데이터 불요, DOM 약결합). 가역성: 높음. 학습: 메뉴별 셀렉터 앵커 관계 정리.

### 대안 B: 자동 시연 (투어가 실제로 행 클릭 → 인스펙터 오픈 → 실버튼 스포트라이트)
- 핵심: 투어가 첫 행을 실제 클릭해 인스펙터를 띄우고 진짜 버튼을 하이라이트. 가장 몰입적.
- 비용: 대. 위험: **높음** — 빈 목록/실데이터 의존, async 대기, 행 선택이 앱 상태 변경, DOM·데이터 변동에 취약. 가역성: 중. 학습: 시연 자동화 난이도.

### 대안 C: 정적 도움말 패널/매뉴얼 (driver.js 비사용)
- 핵심: 메뉴별 도움말 드로어(텍스트+스크린샷). 스포트라이트 대신 읽는 매뉴얼. (기존 manuals-menu 브레인스톰과 일부 중복 가능)
- 비용: 중. 위험: 낮음. 가역성: 높음. 학습: 적음. 단점: "투어 확장" 요청 의도(스포트라이트 체험)와 결이 다름.

### 대안 Z: do nothing
- 전역 5스텝 유지. 신규 구성원이 메뉴별 버튼을 계속 더듬음. 우회책(구두 안내/매뉴얼 문서)으로 부분 해소되나 셀프 온보딩 미흡.

| 항목 | A (스포트라이트+설명) | B (자동 시연) | C (정적 매뉴얼) | Z |
|------|------|------|------|------|
| 비용 | 중+콘텐츠대 | 대 | 중 | 0 |
| 위험 | 낮음 | 높음 | 낮음 | 낮음 |
| 가역성 | 높음 | 중 | 높음 | - |
| 의도 적합 | 높음 | 높음(취약) | 중 | 낮음 |

## 추천 + 근거
**추천: 대안 A** (온디맨드 메뉴별 가이드 — 스포트라이트 + 설명, 자동 조작 X)

- 사용자 의도(투어 확장 + 메뉴별 + 인터랙션/버튼 설명)를 충족하면서, 인스펙터를 "설명"으로 다뤄 실데이터·async 의존을 제거 → 가장 안정적.
- 기존 driver.js·사이드바 `slug`·권한 필터된 `sections`를 그대로 재사용해 인프라 추가 최소.
- 점진 확장 가능 — 인프라(동적 스텝 + 사전 + 앵커) 먼저, 콘텐츠는 메뉴별로 채움.

**기각 B**: 자동 행 클릭은 빈 목록/데이터 변동/상태오염에 취약. 데모용으로는 매력적이나 운영 화면 투어로는 위험 과다. (안정적 데모 환경이 보장되면 일부 핵심 메뉴에 한해 B 전환 가치 있음.)
**기각 C**: 읽는 매뉴얼은 "스포트라이트 체험" 의도와 결이 다름. 단, 버튼 세부가 매우 많은 메뉴는 A의 마지막 스텝에서 C형 도움말로 링크하는 하이브리드 여지.
