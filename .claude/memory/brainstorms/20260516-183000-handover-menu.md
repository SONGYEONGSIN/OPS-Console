# Brainstorm — 인수인계 (handover) 메뉴 (PR-8 epic)

생성: 2026-05-16
관련: 사이드바 mock `handover` slug 활용. 참고 시트 `서비스기록` (1Biglnbic...). UI 참고: settings pattern (일반/알림/표시/보안/통합).

## 배경

운영부가 서비스 인수인계용 long markdown 문서를 시트로 누적해온 자료(`작업이력` 탭, 18 컬럼, 22+ row). 이를 Folio handover 메뉴에서 **3탭** + **settings pattern**으로 정리. services 2511 row 기준 1:1 매핑.

## 사용자 의사결정

| # | 결정점 | 답 |
|---|--------|----|

| 2 | 인수인계 내용 UI 패턴 | **settings pattern** (참고: dashboard 일반/알림/표시/보안/통합 화면). 좌측 카테고리 6 nav + 우측 form |

## 데이터 모델

### handover_records — 인수인계 내용 (service 1:1)

```sql
create table public.handover_records (
  id                   uuid primary key default gen_random_uuid(),
  service_id           uuid not null unique references public.services(id) on delete cascade,
  -- 계약 (2)
  contract_info_md     text,
  contract_data_md     text,
  -- 작업 (7)
  work_basic_md        text,
  work_generator_md    text,
  work_site_md         text,
  work_output_md       text,
  work_rate_md         text,
  work_file_md         text,
  work_etc_md          text,
  -- 정산 (2)
  payment_fee_md       text,
  payment_invoice_md   text,
  -- 연락처 (1)
  school_contact_md    text,
  -- 서류제출 (1)
  docs_md              text,
  -- 기타 (1)
  notes_md             text,
  -- 메타
  author_email         text not null,
  author_name          text not null,
  status               text not null default 'draft'
                         check (status in ('draft','ready','published')),
  created_at           timestamptz not null default now(),
  updated_at           timestamptz not null default now()
);
create index handover_records_service_idx on public.handover_records (service_id);
create index handover_records_status_idx  on public.handover_records (status);
```

**status 의미**:
- `draft`: 작성 중 (입력 시작 미완)
- `ready`: 작성 완료 (인수인계 진행 가능)
- `published`: 진행 완료 (handover_progress.status='completed')

### handover_progress — 인수인계 진행 이력

```sql
create table public.handover_progress (
  id                  uuid primary key default gen_random_uuid(),
  service_id          uuid not null references public.services(id) on delete cascade,
  from_email          text not null,
  from_name           text not null,
  to_email            text not null,
  to_name             text not null,
  status              text not null default 'in_progress'
                        check (status in ('in_progress','completed','cancelled')),
  notes               text,
  confirmed_at        timestamptz,
  created_at          timestamptz not null default now(),
  updated_at          timestamptz not null default now()
);
create index handover_progress_service_idx on public.handover_progress (service_id);
create index handover_progress_to_idx       on public.handover_progress (to_email);
create index handover_progress_created_idx  on public.handover_progress (created_at desc);
```

**RLS** (backup_requests 패턴): authenticated read 모두 / admin·member write / admin delete.

## 폼 동작

- 카테고리 클릭 = URL `?cat=` 만 변경. *form 내용은 모든 카테고리 메모리 보존* (client state)
- 저장 시 `upsertHandoverRecord(serviceId, allFields)` — 14 필드 모두 patch (변경 안 된 필드는 기존 값 유지)
- dirty 추적 — 저장 버튼 enabled/disabled
- 저장 후 status 자동:
  - 14 필드 중 1개 이상 채워짐 → `draft`
  - 모두 채워지면 `ready` (옵션 — 명시 토글로 사용자가 변경 가능하게도 가능)
- 권한:
  - 작성·수정: admin/member 모두 (1차)
  - 조회: 모두

## 리스크

| 리스크 | 완화책 |
|--------|--------|
| 14 sub-field 단일 row — wide table | services와 1:1 매핑이라 자연. 정규화 시점은 sub-field 30+ 늘어날 때 |
| handover_records와 services 동기화 | service 삭제 시 CASCADE. 신규 service는 첫 작성 시 lazy upsert |
| 카테고리 클릭 시 메모리 상태 손실 | client state에서 14 필드 모두 유지. URL은 cat key만 |
| 권한 — 본인 운영 서비스만 vs admin 전체 | 1차는 admin/member 모두 작성 (단순), 후속에 operator-only 정책 추가 |
| 빈 상태 EmptyState | "아직 작성된 인수인계가 없습니다 — 첫 작성을 시작하세요" |
| 시트 22 row import는 PR-C 분리 | 시트의 18 컬럼 → 14 sub-field 매핑 스크립트. PR-A 안정 후 |
