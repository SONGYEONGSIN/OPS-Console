# Brainstorm: 인수인계 기록 작성폼 카테고리 레일을 운영가이드 nav 표준으로

## 의도

- **산출물**: `/dashboard/handover/[serviceId]` 기록 작성폼의 좌측 카테고리 레일(`HandoverCategoryRail`)을 운영가이드 좌측 nav(`OpsGuideNav`)와 동일한 선택/호버 표준으로 재스타일.
- **사용자**: 운영자 — 인수인계 기록 작성 시 카테고리 전환. 운영가이드와 동일한 시각 언어로 학습 비용 제거.
- **트리거 (왜 지금)**: #846·#848에서 "목록/메뉴 항목형 선택·호버 표준 = 운영가이드 좌측 nav" 로 확정됨(CLAUDE.md 인터랙션 표준). HandoverCategoryRail은 구표준(잉크 반전 `bg-ink text-cream` + `hover:bg-washi`)으로 남아 있어 불일치. 사용자가 직접 통일 제안.
- **성공 기준**: 레일 선택 항목이 `border-vermilion bg-vermilion/10 text-vermilion`(◉ 마커), 비선택 호버가 `hover:bg-line-soft`. 기존 기능(○◐● 진행 마크, n/m 카운트, 하단 합계) 유지. variant 테스트 GREEN.

## 제약

- **기술**: 진행 상태 마크(○◐●)와 카운트는 운영가이드 nav에는 없는 요소 — 표준 색만 이식하고 진행 표시는 유지해야 함.
- **비즈니스**: 머지 = Vercel 프로덕션 즉시 배포. 사용자 라이브 검증 관례.
- **코드베이스**: 인터랙션 표준은 CLAUDE.md Design System 섹션에 명문화됨. `HandoverCategoryRail.tsx`(68줄) 단일 컴포넌트 + `[serviceId]/__tests__` 존재. 버튼/토글 호버는 이 표준 적용 금지(별도) — 레일은 '메뉴 항목형'이므로 적용 대상.

## 추천 + 근거

**추천: 대안 A** — 사용자 의도(시각 통일)는 선택/호버 색 표준 이식만으로 충족된다. 진행 마크·카운트·하단 합계는 작성폼 고유 가치라 유지. ◉/· 마커는 ○◐● 진행 마크가 이미 그 자리를 차지하므로 도입하지 않음(정보 중복).

**기각된 B**: 진입 경로가 목록 → 서비스 클릭 단일 흐름이라 URL 딥링크(?cat=) 실익이 낮고, 자동저장 중 URL 변경 상호작용 리스크만 추가. 카테고리 공유 링크 요구가 생기면 그때 전환 가치 있음.
